AutoRoute — Technical Overview

This document is a complete technical reference for the AutoRoute fuel dispatch system. It covers how every part of the system works — from the Excel files that feed it with data, to the routing algorithm that assigns drivers, to the web dashboard dispatchers use every day. Whether you are a developer joining the project, an administrator setting it up for the first time, or simply someone trying to understand how the system makes decisions, this guide is written for you.

v1.0 · June 2026 Python 3 · FastAPI 0.115 Next.js 14 · TypeScript Supabase · PostgreSQL Google Maps Routes API Render · Vercel

00 What Is AutoRoute?

AutoRoute is a software system that automatically plans daily fuel delivery routes for a fleet of tanker truck drivers. Instead of a dispatcher spending hours manually matching drivers to loads, the system takes in that day's delivery orders, checks which drivers are available and certified, calculates realistic drive times using Google Maps, and produces an optimized schedule — in seconds.

The Problem It Solves

A fuel distribution company receives hundreds of delivery orders per day. Each order (called a load or identified by a CE ID) specifies a customer site that needs fuel, which terminal (fuel loading rack) to pick up from, what products are needed (Regular, Diesel, etc.), how many gallons, and sometimes a delivery window the customer expects to be served within.

Dispatchers previously had to manually match each load to an available driver, taking into account: Does the driver have an access card for that terminal? Are they certified to operate the pump at that site? Are they restricted from certain customers? Will the driving time fit within their shift? Will they arrive on time for the delivery window? Doing this for 15+ drivers and 100+ loads every morning is error-prone and time-consuming.

AutoRoute automates this entire process. It enforces all the same rules the dispatcher would check manually, then applies an optimization algorithm to minimize wasted empty miles across the entire fleet.

How a Typical Day Works

Here is the end-to-end picture of a typical dispatch day, in plain language:

  1. Overnight / early morning: The sync script (running on a PC at the office) reads the latest Excel files and pushes all updated load orders, driver schedules, and terminal access data into the Supabase database.
  2. Morning: A dispatcher opens the AutoRoute dashboard in their browser. They check which drivers are confirmed as working that day, toggling attendance as needed.
  3. Run Dispatch: The dispatcher clicks "Run Dispatch" for today's date. The routing engine runs in seconds, assigning loads to drivers, sequencing stops, and calculating ETAs.
  4. Review the Board: The dispatch board shows each driver's assigned loads as cards in columns. The dispatcher can drag cards to resequence, see which loads are unassigned and why, and make manual adjustments.
  5. Mid-day reroute: As drivers start their routes, their load statuses update in real time via the sync. If a new urgent load comes in, the dispatcher runs "Reroute" — the engine locks any loads already in progress and re-optimizes the rest.
  6. End of day: All delivered loads show as status 26 (Delivered) on the board. The full day's run is stored in the database for audit purposes.

01 System Architecture

AutoRoute is built as four distinct layers, each with a clear responsibility. Data flows in one direction: from the source Excel files → into the database → through the API → to the browser. No layer needs to know how the others work internally, which makes each part easy to replace or upgrade independently.

📊 Source Data
Excel files on a local PC, auto-refreshed by ODBC from the company's dispatch system
↓  sync/sync.py — reads Excel, pushes to Supabase every 5 minutes
🗄️ Database — Supabase (PostgreSQL)
Single source of truth · 12 tables · Row-Level Security · accessible from anywhere
↙ backend reads via supabase-py (service key) frontend reads via supabase-js (anon key) ↘
⚙️ Backend API
Python · FastAPI · hosted on Render
🖥️ Frontend Dashboard

Layer 1 — Source Data (Excel + ODBC)

The company's dispatch system is a legacy application that exports data to Excel files via ODBC (a database connection protocol). These Excel files are the authoritative source for loads, driver schedules, terminal access cards, and site details. AutoRoute does not connect to the legacy system directly — it reads these files and works from there.

This approach means AutoRoute can be set up without IT access to the core dispatch database, and the legacy system continues operating unchanged alongside it.

Layer 2 — Database (Supabase)

Supabase is a hosted PostgreSQL database with a built-in REST API, authentication, and row-level security. It acts as the central hub that all parts of the system read from and write to. The sync script writes to it from the local PC; the FastAPI backend reads from it on every dispatch run; the frontend talks to it directly for authentication.

Choosing Supabase means there is no infrastructure to manage — backups, scaling, and connections are all handled by the service. The anon (public) key allows the frontend to authenticate users safely; the service role key (used only on the backend) bypasses row-level security for admin database operations.

Layer 3 — Backend API (FastAPI on Render)

The Python FastAPI backend is the brain of the system. It hosts the routing engine, validates all business logic, enforces authentication on every request, and exposes a clean REST API to the frontend. It runs as a single-process web server on Render. There are no background workers or job queues — every dispatch run is kicked off by a direct HTTP request and runs to completion synchronously (though results are persisted in the background so the user is not waiting).

Layer 4 — Frontend (Next.js on Vercel)

The frontend is the dispatcher's day-to-day interface. It is a React application built with Next.js that renders the dispatch board, driver list, load management screens, and all admin tools. It is a "thin" client — it does not contain any routing logic itself. All it does is fetch data from the API, display it, and send user actions (run dispatch, toggle attendance, drag a card) back to the API.

Key Design Decisions

  • No microservices. A single FastAPI process handles everything. This keeps deployment, debugging, and local development simple. If the system needs to scale, the Render plan can be upgraded; there is no distributed system complexity to manage.
  • Supabase as the integration bus. Both the sync script (running on a PC) and the API (running on Render) share the same database. This eliminates the need for any message queues, webhooks, or direct connections between components.
  • Stateless routing engine. The engine loads all the data it needs from Supabase at the start of each run, computes the result entirely in memory, and writes it back. If the server restarts mid-run, no state is lost — the next run starts fresh. This makes the system very easy to reason about and debug.
  • Frontend uses two Supabase clients. Authentication (login, session management) is handled directly between the browser and Supabase Auth. Business data (loads, drivers, dispatch results) flows through the FastAPI backend. This cleanly separates user identity from application logic.

02 Data Flow

Understanding how data moves through the system is essential before diving into any individual component. There are three distinct data flows, each triggered by a different event: the background sync that keeps the database current, the dispatch run that produces assignments, and the board load that paints the dispatcher's screen.

Flow 1 — Sync (Runs Every 5 Minutes in the Background)

The sync script is a Python program running on a local Windows PC at the office. It watches a folder of Excel files and, every five minutes, reads each file and pushes its data into Supabase. This is the only way data gets into the database — the API never writes load orders or driver schedules itself, it only reads them. 

Excel file on disk
  → copied to temp file (in case ODBC has the file locked)
  → pandas reads the Excel sheet(s)
  → transformer function normalizes column names, coerces types, deduplicates
  → Supabase upsert in 500-row chunks
  → result (rows upserted, duration, any error) written to sync_log table

The load_details table is special: the loads file has two sheets that must be combined before upserting. The Main sheet contains order data (what to deliver, where, when), and the Driver & Terminal sheet contains live status updates from the legacy system (which driver is currently assigned, which terminal they loaded from). These two sheets are merged on ce_id + product_name before any data reaches the database.

To keep syncs fast, only loads from the past 60 days are processed. Historical data already in Supabase is left untouched.

Flow 2 — Dispatch Run (On-Demand, Triggered by a Dispatcher)

When a dispatcher clicks "Run Dispatch" in the browser, the frontend sends a POST /api/dispatch/run request to the backend API. The API then orchestrates the entire routing process: 

POST /api/dispatch/run  { dispatch_date: "2026-06-25", reroute: false }
  ├─ Load yards, terminals, sites from Supabase (reference data)
  ├─ Load active drivers for the date (attendance_expected = 1)
  │    └─ Attach each driver's terminal access cards + site restrictions
  ├─ Load all loads for dispatch_date ± 1 day
  │    └─ Resolve terminal IDs by name; infer missing terminals from history
  ├─ Clear the in-process travel time cache
  ├─ Run RoutingEngine in a thread pool (so async event loop stays free)
  │    ├─ Phase 1: Greedy assignment pass
  │    ├─ Phase 2: Retry pass (timing-only failures)
  │    └─ Phase 3: 2-opt local search improvement
  ├─ Return result JSON to the frontend immediately
  └─ Persist dispatch_results + unassigned_loads to Supabase [background]

The engine result is returned to the frontend immediately — the dispatcher sees the board update without waiting for the database write to complete. The background persistence task runs after the HTTP response is already sent.

Flow 3 — Board Load (Every Time the Date Changes or Page Loads)

The dispatch board is not just showing engine output — it combines three different data sources into a single unified view. When the frontend requests the board for a given date, the API: 

GET /api/dispatch?dispatch_date=2026-06-25
  ├─ dispatch_results   — rows written by the routing engine for this date
  ├─ pre_assigned loads — loads with status > 1 (already in motion per ODBC),
  │                       matched to driver names from driver_schedules
  ├─ unassigned_loads   — loads the engine could not assign, with reasons
  └─ load_details       — full metadata for all loads (paginated, up to 1000/page)

  Frontend merges these four sources:
  ├─ Pre-assigned loads appear at the top of their driver's column
  ├─ Engine-assigned loads appear below, sorted by route_sequence
  └─ Unassigned loads appear in a collapsible panel below the board

This design means the board shows a complete picture of the day whether or not the engine has been run yet. If a driver's loads already started before dispatch ran, they still appear correctly on the board — pulled from the ODBC data rather than engine output.

03 Database Schema

The database has 12 tables grouped into four categories: reference data (the geographic and operational master data), driver data (schedules, access, and restrictions), load data (the delivery orders), and engine output (the results produced by each routing run). All tables live in a single Supabase project.

Reference Tables — The Geographic Master Data

These three tables are relatively static. They describe the physical locations in the system and are updated by the sync script from Excel, but rarely change day-to-day.

TablePrimary KeyDescription
yard_locations yard (text) Each yard where drivers start and end their shift. Stores address, city, state, and lat/lon coordinates used to calculate drive-back-to-yard time.
terminal_locations terminal_id (int) Fuel loading racks (terminals). Stores lat/lon for routing, plus an is_diesel_wet flag (0 or 1) that triggers the product sequencing rule described in the routing engine section.
site_details site_id (int) Customer delivery locations. Stores lat/lon, the customer group name (used for group-level driver restrictions), and a pump_certified flag that controls which drivers are eligible for that site.

Driver Tables — Schedules, Access, and Restrictions

These three tables define everything the routing engine needs to know about a driver on a given day. They are updated daily by the sync script from the driver schedule Excel file, and can also be edited manually through the dashboard.

TablePrimary KeyDescription
driver_schedules record_id (int) One row per driver per shift date. Contains the driver's name, yard, start time, board location (TX-AM etc.), shift length, pump training status, and most importantly attendance_expected — a 0/1 flag that dispatchers can toggle to mark a driver as present or absent for the day. Only drivers with attendance_expected = 1 are included in the dispatch run.
driver_terminal_cards (driver_id, terminal_id) A junction table that records which fuel terminals a driver has a valid access card for. A driver can only be assigned a load at a terminal if a row exists here for their driver_id + that terminal_id. This is a hard constraint in the routing engine — no card, no assignment.
driver_restrictions id (serial) Per-driver blacklists. Restrictions can be either site-level (a specific site_id the driver cannot deliver to) or customer-group-level (an entire customer group the driver cannot serve). Reasons might include past incidents, customer preferences, or equipment incompatibilities.

Load Table — The Delivery Orders

load_details is the most important table. It holds every fuel delivery order. Its primary key is (ce_id, product_name) — a composite key — because a single order can require multiple products (e.g. a station might need both Regular gasoline and Diesel in the same delivery). Each product in the order gets its own row in this table, but they all share the same CE ID.

The routing engine groups rows by CE ID and treats them as a single load. The combined set of products on a load determines whether diesel-wet sequencing rules apply.

Key columns include: delivery_date, site_id, terminal_id, terminal_name, product_name, gross_gallons, window_start / window_end (the customer's requested delivery window, or NULL for anytime deliveries), load_status (see below), and first_name / last_name (the driver currently assigned in the legacy ODBC system, used to populate the pre-assigned section of the dispatch board).

Each load has a numeric status that represents where it is in its lifecycle. This status is the key signal the routing engine and board use to decide how to handle each load:

Engine Output Tables — What the Routing Run Produces

After each dispatch run, the engine writes its results into three tables. These are replaced on every new run for the same date (the previous run is marked inactive, not deleted, for audit purposes).

Row-Level Security (RLS)

Supabase's row-level security lets you define access policies directly in the database, so even if someone obtained the anon key, they could only see data they are authorized to see. RLS is enabled on the five most sensitive tables: app_users, dispatch_results, unassigned_loads, dispatch_runs, and driver_restrictions.

The policies are straightforward: any authenticated user can read dispatch data; only users with the administrator role can read or modify user records. The backend API uses the service role key which bypasses RLS entirely — but this key is never exposed to the browser.

04 Sync Pipeline

The sync pipeline is what keeps AutoRoute's database current. It is a Python script that runs in the background on a local PC, reads six Excel files, and pushes their contents into Supabase every five minutes. Without this sync running, the routing engine would operate on stale data.

Filesync/sync.py

Interval300 seconds by default (configurable via SYNC_INTERVAL_SECONDS in .env)
Dependenciespandas, openpyxl, supabase-py, python-dotenv

Why Excel Files Instead of a Direct Database Connection?

The company's legacy dispatch system uses an internal proprietary database. Rather than requesting IT access to that database (which would require corporate approval, VPN configuration, and ongoing maintenance), AutoRoute reads the same Excel exports that dispatchers already use day-to-day. These files are automatically kept up-to-date via ODBC (a standard database connectivity protocol built into Windows), so the sync effectively piggybacks on an existing workflow.

This approach has one practical gotcha: ODBC sometimes keeps an Excel file locked open while it is refreshing it. To handle this, the sync script copies each file to a temporary location before reading it — so even if the source file is locked, the sync still gets the latest data.

The Transformer Pipeline — How Raw Excel Becomes Clean Database Rows

Excel files from ODBC are messy. Column names have inconsistent casing, numeric values arrive as floats when the database expects integers, dates might be formatted as Excel serial numbers or as text strings, and some columns the database doesn't have appear in the file. Each table has a dedicated Python transformer function that normalizes all of this before the data reaches Supabase:

  1. Normalize column names — all headers lowercased and stripped. Renamed to match DB schema (e.g. BillOfLading_ID → ce_id).
  2. Coerce numeric types — non-numeric values (like "T-75-TX-2664" in a terminal_id column) are converted to None rather than crashing.
  3. Parse dates and datetimes — handles both Excel's internal serial number format (days since 1899-12-30) and ISO string format. All datetimes are stored timezone-naive in the database to avoid comparison issues in the engine.
  4. Apply domain rules — e.g. clamp pump_certified to exactly 0 or 1 (any other numeric value becomes 0); derive board_location by concatenating division_prefix and default_shift_name.
  5. Drop extra columns — any column not present in the DB schema is removed. This protects against ODBC adding new columns that would cause the upsert to fail.
  6. Deduplicate on primary key — within the same Excel file, if the same CE ID + product appears twice, keep only the last occurrence. This prevents batch ON CONFLICT errors.

Upsert Strategy and Load Protection

Records are pushed to Supabase using "upsert" — insert if new, update if already exists. Data is sent in chunks of 500 rows to stay within Supabase's request size limits.

There is one important protection for load data: if a load row already exists in Supabase with load_status = 1 (meaning a dispatcher has manually marked it as "ready to route"), the sync skips that row and does not overwrite it with whatever the ODBC file says. This prevents the source system from accidentally resetting loads that are staged for routing.

Error Handling and Reliability

If one table's sync fails (e.g. a malformed Excel file), the error is logged to sync_log and the script moves on to the next table. A single bad file does not stop all other tables from syncing. The sync status indicator in the dashboard header shows the timestamp of the last successful sync, so dispatchers can immediately tell if the data is fresh or stale.

The backend is a Python web server built with FastAPI. It is the only component that runs the routing engine, and it is the gatekeeper for all data modifications — every action a dispatcher takes in the browser goes through this API. All endpoints require a valid Supabase JWT, so unauthenticated requests are rejected before any business logic runs.

How Authentication Works

When a user logs into the dashboard, Supabase Auth returns a JWT (JSON Web Token). The frontend stores this token in memory and attaches it to every API request as a Bearer token. The FastAPI backend's verify_token dependency function calls supabase.auth.get_user(token) to validate the token — if invalid or expired, the request is rejected with a 401. This means the backend never stores passwords or manages sessions; that is all handled by Supabase.

The Dispatch Run in Detail

The most important endpoint is POST /api/dispatch/run. It accepts two parameters: dispatch_date (a date string like 2026-06-25) and reroute (true or false).

  • Normal dispatch (reroute: false): Used at the start of the day. Routes all loads with status ≤ 1 (unscheduled). Any previous dispatch results for the same date are marked inactive and replaced.
  • Reroute (reroute: true): Used mid-day when something has changed — a new urgent load came in, a driver called in sick, or a previous assignment turned out to be wrong. The engine first "seeds" the solution with loads that are already in motion (status 12 = en route to rack), locking them to their current driver. Then it re-routes all remaining unstarted loads around those locked assignments.

The routing engine itself is synchronous Python code — it does not use async/await. To run it inside the async FastAPI server without blocking the event loop, it is dispatched to a thread pool via asyncio.run_in_executor(None, engine.run). This also ensures the engine's own travel time lookups can create fresh event loops internally, which is needed for the Google Maps API calls to work correctly from a worker thread.

06 Routing Engine

The routing engine is the heart of AutoRoute. Given a set of drivers and loads for a day, it produces the best possible assignment of loads to drivers, taking into account dozens of real-world constraints and trying to minimize wasted empty miles across the entire fleet. This section explains how the algorithm works, step by step.

Filebackend/engine/routing_engine.py
ClassRoutingEngine
AlgorithmGreedy assignment → retry pass → 2-opt local search
Score functiontotal_empty_miles − total_loaded_miles (lower = better; rewards loaded driving, penalises deadhead)

What "Optimization" Means Here

The engine is trying to minimize empty miles — miles driven without fuel on board (deadhead). Every mile a driver spends driving to a terminal without a load, or returning to the yard at the end of the day, is a cost. The algorithm scores each possible assignment using empty_miles − loaded_miles: a lower (more negative) score means more loaded driving relative to empty driving, which is better.

This is a variant of the classic Vehicle Routing Problem (VRP), which is computationally hard in the general case. Rather than trying every possible combination (which would take hours for 15 drivers and 100 loads), the engine uses a fast greedy heuristic to get a good initial solution, then improves it through local search.

Input: What the Engine Receives

Before the engine runs, the data loader fetches everything it needs from Supabase:

  • Drivers — only those with attendance_expected = 1 for the dispatch date. Each driver object carries their yard location, start time, max shift hours, pump training status, terminal access card set, and restriction sets.
  • Loads — all loads for dispatch_date ± 1 day, filtered to status ≤ 1. Each load has its site and terminal resolved from the reference dictionaries. Loads with missing site or terminal data are immediately placed in the unassigned list with an explanatory reason.
  • Reference dictionaries — sites, terminals, and yards, keyed by their IDs for fast lookup.

Phase 1 — Greedy Assignment

The first phase processes loads one at a time in priority order. The goal is to quickly produce a complete, feasible solution even if it is not yet optimal.

Load priority order:

  1. Loads with a specific delivery time window (timed deliveries) come before loads with no window (anytime deliveries). A gas station that needs fuel between 8 AM and 10 AM is harder to schedule than one with no preference — it gets first pick of drivers.
  2. Among timed loads, earlier delivery dates come first.
  3. Among same-date timed loads, earlier window start times come first.

For each load: Drivers are tried in order of least current work (fewest stops already assigned, then earliest finish time — so the lightest drivers get first shot at new work). For each driver, every possible insertion position in their existing route is tried (before their first stop, between stops, after their last stop). The position that produces the best score is accepted — but only if the full route simulation passes all constraints.

If no driver can take the load at any position without violating a constraint or shift limit, the load is added to the unassigned list along with the most informative failure reason.

Phase 2 — Retry Pass

After the greedy pass, some loads may be unassigned simply because every driver's shift was slightly too full at the time that load was tried — not because of any hard rule violation. The 2-opt pass in Phase 3 often reshuffles routes enough to free up capacity. But rather than running 2-opt blind and hoping it frees the right slots, the retry pass gives timing-only failures a dedicated second chance.

Loads that failed exclusively with the reason "Shift time exceeded" are collected and retried against drivers sorted by finish time (lightest workload first). Any load that finds a slot now is accepted and added to the solution before 2-opt begins.

Phase 3 — 2-opt Local Search

After Phases 1 and 2 produce an initial complete assignment, the 2-opt pass iterates over all pairs of drivers and tries to improve the solution by making small swaps and moves:

A
SWAP — Exchange one load between two drivers

For every pair of drivers A and B, try swapping every non-locked load on A's route with every non-locked load on B's route. If the combined score of both routes improves by more than 0.01 miles, accept the swap and update both routes. Then continue checking this driver pair with the updated routes.

B
MOVE — Relocate one load from one driver to another

For every non-locked load on driver A's route, try inserting it into every possible position on driver B's route (as long as B has fewer than 4 stops). If moving the load reduces the combined empty miles of both routes, accept the move. This is particularly effective when one driver is far from a load's terminal and another driver is nearby.

The loop repeats — trying all pairs again with the updated assignments — until a full pass produces no improvements, or 200 iterations are reached (a safety cap to prevent runaway compute on unusual data). In practice, the engine converges in 3–10 iterations for a typical day.

Loads with locked statuses (status 12 = en route to rack, status 20 = at rack) are never touched by the 2-opt pass. You cannot pull a driver off a load they are already physically loading.

Route Simulation — How the Engine Checks Feasibility

Before accepting any assignment, the engine simulates the full timeline for a driver with their proposed list of stops. This _simulate_route function is called thousands of times per dispatch run. It walks through each stop in sequence: 

Starting at driver's yard at shift start time:

  For each stop (load assignment):
    1. Calculate drive time from current position to terminal
       → Uses Google Maps API (with 20% tanker multiplier) or haversine fallback
    2. Arrive at terminal. Check: does driver arrive before shift end?
    3. Load service time: 30 minutes at terminal
    4. Calculate drive time from terminal to delivery site
    5. Arrive at site. Apply delivery window logic:
       ├─ If arriving more than 2 hours early → wait at staging area
       ├─ If arriving up to 1 hour late → allowed (grace period)
       └─ If arriving more than 1 hour late → simulation FAILS (window missed)
    6. Unload service time: 45 minutes at site
    7. Update current position and time to end of this stop

  After all stops:
    8. Calculate drive time from last site back to yard
    9. Check: does driver return before shift end?
       If not → simulation FAILS (shift exceeded)

Returns: complete DriverRoute object with ETAs for every stop,
         or None if any constraint was violated

If _simulate_route returns None, that insertion is infeasible. The engine tries the next position, the next driver, until all options are exhausted.

Hard Constraints — Rules That Can Never Be Violated

These are checked before simulating a route. If any hard constraint fails, the driver is immediately skipped for that load:

ConstraintWhat It ChecksWhere in Code
Terminal card access The load requires terminal X. Does this driver have a row in driver_terminal_cards for terminal X? If not, they cannot pick up the load. _check_driver_eligible
Pump certification The delivery site has pump_certified = 1. Does this driver have pump_trained = 1? Uncertified drivers cannot deliver to pump-required sites. _check_driver_eligible
Site restriction Is the load's site_id in this driver's restricted_site_ids set? If so, the driver cannot be assigned to this site. _check_driver_eligible
Customer group restriction Is the site's customer group name in this driver's restricted_customer_groups set? If so, blocked even if the specific site is not individually restricted. _check_driver_eligible
Diesel-wet sequencing If the load is at a diesel-wet terminal, the previous load on this driver's route must contain diesel and must NOT contain gasoline or dyed fuel. This reflects real tanker compartment contamination rules. _check_diesel_wet_sequence
Shift time The driver must be able to complete all stops and return to their yard before their shift ends. _simulate_route (returns None)
Delivery window If the load has a time window, the driver must arrive within the tolerance (up to 2 hours early, up to 1 hour late). _simulate_route (returns None)
Max 4 stops per driver A driver's route cannot exceed 4 loads. This is a practical limit based on typical shift lengths. Greedy loop guard

Unassigned Reason Priority

When a load cannot be assigned, the engine has often accumulated multiple failure reasons — one per driver tried. For example, Driver A failed due to "no terminal access", Driver B failed due to "shift exceeded", and Driver C failed due to "pump cert required". Showing all reasons would be confusing.

Instead, the engine picks the single most informative reason using a fixed priority ranking. Hard structural blockers (terminal access, pump certification, site restrictions) rank higher than soft timing failures. So if even one driver was blocked by a terminal access issue, that becomes the displayed reason — because fixing it (granting the driver a terminal card) would potentially unlock the load, whereas "shift exceeded" just means a driver was busy.

07 Geo & Travel Time

Accurate travel time estimates are critical for route feasibility. A route that looks fine on a map might actually violate a driver's shift limit once real-world traffic is taken into account. AutoRoute uses Google Maps for live traffic-aware estimates, with a built-in fallback and intelligent caching to control API costs.

Filebackend/engine/geo.py
Primary fnget_travel_mins_sync(lat1, lon1, lat2, lon2, departure_epoch)
FallbackHaversine (straight-line) distance at 50 mph constant speed
MultiplierAll results ×1.20 — accounts for tanker truck speed vs. passenger car baseline

Distance Calculation — Haversine Formula

The haversine formula calculates the great-circle distance between two latitude/longitude points on the Earth's surface. It is used as both a fallback travel time estimate (at 50 mph) and to calculate loaded and empty mileage figures shown on the dispatch board.

Earth radius used: 3958.8 miles. The function is haversine_miles(lat1, lon1, lat2, lon2).

💡

Haversine gives straight-line distance, not road distance. Road distance is typically 15–25% longer. This is partially offset by the 20% tanker multiplier — so haversine-only mode is a rough but usable fallback, not a precise estimate.

Google Maps Routes API

When a Google Maps API key is configured, the engine uses the Routes API v2 (computeRoutes endpoint) to get traffic-aware driving durations. The request specifies DRIVE mode, TRAFFIC_AWARE routing preference, and avoids ferries. It passes the expected departure time so Google can use traffic predictions appropriate for that time of day.

ParameterValue / Behavior
Endpointhttps://routes.googleapis.com/directions/v2:computeRoutes
Travel modeDRIVE · TRAFFIC_AWARE · ferries avoided
Field maskroutes.duration,routes.distanceMeters — only request what's needed to minimize response size and billing
Departure timePassed as an RFC3339 UTC timestamp. If the departure time is in the past (which can happen on a reroute), it is omitted and Google uses current-time traffic.
Timeout8 seconds per call. If Google does not respond in time, the call fails gracefully and the haversine fallback is used for that leg.

The 20% Tanker Multiplier

Google Maps estimates driving times based on typical passenger cars. A fully loaded fuel tanker is a heavy commercial vehicle that takes longer to accelerate, must maintain greater following distances, and cannot match car speeds on ramps and surface streets. Based on real-world observation of actual vs. estimated times, AutoRoute applies a flat 20% increase to every travel time estimate — whether it came from Google Maps or haversine.

This multiplier is stored as a single constant (TANKER_TRAVEL_MULTIPLIER = 1.20) in geo.py. If operational data shows the figure should be adjusted, changing this one value updates all route simulations.

Caching — Staying Within Google's Free Quota

A typical dispatch run with 15 drivers and 4 stops each involves about 120 travel legs (yard → terminal, terminal → site, for each stop). Without caching, that would be 120 Google Maps API calls per run. Google's free tier is 100 requests/day.

The solution is an in-process cache (a Python dictionary) that stores results for the lifetime of the dispatch run. The cache key is: 

(round(lat1, 4), round(lon1, 4), round(lat2, 4), round(lon2, 4), departure_epoch // 900)

Coordinate rounding to 4 decimal places gives ~11 meter precision. Since terminals and sites have fixed locations, two drivers heading to the same terminal from the same area will get the same cache entry, even if their exact starting coordinates differ slightly.

15-minute departure time buckets (epoch // 900) means drivers leaving within the same 15-minute window share a cached result. Traffic does not meaningfully change within 15 minutes on Texas highways, so this is a safe approximation.

In practice, most terminal→site legs repeat across many drivers (e.g. 8 drivers all going to the same Shell station from the same Dallas rack). With caching, a typical run makes only 20–30 real API calls — well within the free quota. The cache is cleared at the start of each new dispatch run so stale results from a prior run never carry over.

Threading Considerations

The routing engine is synchronous code running in a thread pool executor (not the FastAPI async event loop). The Google Maps call is an async HTTP request. To bridge this, get_travel_mins_sync creates a brand new event loop (asyncio.new_event_loop()), runs the async call to completion inside it, then closes the loop. This is intentional — reusing the FastAPI event loop from a worker thread would cause errors in Python 3.10+.

Dispatch Board Tab — The Core Feature

The dispatch board is the most complex component in the application. It shows a kanban-style view of drivers as columns and their assigned loads as cards within each column.

When the tab loads (or the date changes), it calls GET /api/dispatch and builds the board from the combined response. Drivers are grouped into sub-tabs by board location (TX-AM, TX-PM, FW-AM, FW-PM, ET-AM). Within each board location, drivers are sorted alphabetically by last name.

Each load is rendered as a LoadCard showing:

  • Customer name, site name, and city
  • Delivery time window (or "Anytime")
  • Terminal name and product type
  • ETA (if assigned by the engine)
  • CE ID and order number
  • A color-coded status dot indicating the load's current lifecycle state
  • A sequence number badge (1, 2, 3, 4) showing where the load falls in the driver's route

Key dispatcher interactions:

  • Run Dispatch — Calls POST /api/dispatch/run with reroute: false. Board refreshes automatically when the run completes.
  • Reroute — Calls the same endpoint with reroute: true. Loads in progress are locked; others are re-optimized.
  • Drag-and-drop resequencing — Load cards are draggable using @dnd-kit. Dragging a card within a driver's column reorders it; dragging it to another driver's column reassigns it. On drop, the frontend updates the board optimistically and calls PATCH /api/dispatch/{ce_id}/resequence to persist the change.
  • Undo — Before each Run or Reroute, the current board state is saved to a local history array. Clicking Undo restores the board to its previous local state (frontend-only; does not undo database writes).
  • Unassigned panel — Click the red "✗ N Unassigned" button to expand a panel listing every load the engine could not place, along with the reason why.
  • Search — Filter visible loads by CE ID, site name, or order number. Helpful on busy boards with 50+ loads.

All calls from the frontend to the FastAPI backend go through a single typed wrapper module. 09 Authentication & Authorization

AutoRoute uses Supabase Auth for user identity. Authentication is handled entirely by Supabase — the application does not store passwords, implement session logic, or manage JWTs directly. Authorization (what a logged-in user is allowed to do) is enforced at two levels: the frontend hides admin-only UI elements, and the database enforces row-level security policies as a safety net.

How Login Works — Full Token Flow

Here is exactly what happens when a dispatcher signs in:

  1. User enters email + password in the login form and clicks "Sign in".
  2. The frontend calls supabase.auth.signInWithPassword({ email, password }) directly — this request goes to Supabase, not to the FastAPI backend.
  3. Supabase validates the credentials and returns a JWT access token and a refresh token.
  4. The useAuth hook detects the new session via supabase.auth.onAuthStateChange and calls fetchAppUser, which queries the app_users table to retrieve the user's full name and role.
  5. The user is redirected to /dashboard. From this point on, every API call attaches the JWT as an Authorization: Bearer header.
  6. On the backend, the verify_token FastAPI dependency intercepts every request, calls supabase.auth.get_user(token) with the anon key to validate the JWT, and injects the user object into the route handler. If the token is missing, expired, or tampered with, a 401 is returned immediately.

The Two Supabase Keys — And Why Both Exist

Supabase provides two API keys for every project, and AutoRoute uses both for different purposes:

  • Anon key (SUPABASE_ANON_KEY) — A public-safe key that is embedded in the frontend code (prefixed NEXT_PUBLIC_). It is safe to expose because Supabase's row-level security policies govern what this key can access. The backend also uses this key specifically for get_user(token) calls — validating a JWT does not require elevated privileges.
  • Service role key (SUPABASE_SERVICE_KEY) — A secret key that bypasses all row-level security. Used only in the backend API and sync script, never exposed to the browser. Required for operations like upserting thousands of sync rows or deleting old dispatch results.

Role Model

There are exactly two roles, stored in app_users.role:

RoleWhat This User Can Do
user View all tabs (Dispatch Board, Loads, Drivers, Terminal Access, Restrictions). Run dispatch and reroute. Toggle driver attendance. Add/remove terminal access records and restrictions. Cannot see the Users tab.
administrator Everything a user can do, plus: view and manage the Users tab (invite users, change roles, activate/deactivate accounts). Shown with an "ADMIN" badge in the header.

Role enforcement happens in two places: the frontend hides the Users tab for non-admins, and the Users tab component itself checks the role on render and shows "Access denied" as a fallback if the frontend check is somehow bypassed. Database-level RLS adds a third enforcement layer.

UNLOAD_SERVICE_MINS geo.py 45 Time the engine assumes a driver spends at the delivery site unloading. Includes setup and paperwork time.
EARLY_ALLOWANCE_MINS routing_engine.py 120 How early a driver may arrive before a customer's delivery window opens. 120 minutes = 2 hours. Driver will wait at staging if they arrive this early.
LATE_ALLOWANCE_MINS routing_engine.py 60 How many minutes after the window closes a driver is still considered "within tolerance". Does not cause rejection, but the load will show as running late.
REJECT_LATE_MINS routing_engine.py 60 If a driver would arrive more than this many minutes after the window closes, the simulation fails and the assignment is rejected. Prevents assigning loads with missed windows.
MAX_OPT_ITERS routing_engine.py 200 Maximum number of 2-opt improvement iterations before the engine gives up and returns the current best solution. A safety cap to prevent unusually long runs on degenerate data.

13 Troubleshooting

This section covers the most common issues encountered during setup and daily operation. For each symptom, a likely cause and concrete fix are provided. If an issue is not listed here, check the Render logs (backend), Vercel deployment logs (frontend), and the sync_log table in Supabase (sync script).

Adding New Users

New users cannot self-register — they must be invited by an administrator. The process is:

  1. Go to Supabase → Authentication → Users → Invite User and enter the new user's email address. Supabase sends them an invite email with a link to set their password.
  2. Once they click the link and set a password, their account exists in auth.users but they will not appear in the dashboard's Users tab yet — that table requires an app_users row.
  3. An administrator must create the app_users row. This can be done from the Users tab in the dashboard (after the user has signed in at least once), or via SQL:
-- Run this in Supabase SQL Editor after the user has completed sign-up
INSERT INTO app_users (id, email, full_name, role)
SELECT id, email, 'First Last', 'user'
FROM auth.users
WHERE email = 'newuser@company.com';

-- To make them an administrator instead, change 'user' to 'administrator'

After the app_users row is created, the user can log in normally and their role will be applied.

Future: OneDrive / O365 Integration

The current sync script reads Excel files from a local file system path. This means the sync must run on the same PC that has access to those files. A natural next step, when Office 365 permissions are available, is to read the files directly from SharePoint or OneDrive via the Microsoft Graph API.

Implementing this would require only a small change to the sync_table() function in sync/sync.py — replacing the file-system read with an authenticated HTTP download from the Graph API. The transformer pipeline, upsert logic, and all downstream behavior would remain exactly the same. The sync could then run from any machine with internet access (or from a cloud function), rather than requiring a dedicated office PC.
